Graph Representation Eval

How well do different textual formats communicate software architecture topology to an LLM?
Source code · Follow-up: abstract graphs

The Prompt

I'd like you to investigate the question of how the following methods of communicating graph structures to an LLM compare. I'd like you to do this via formal evals, using the techniques described in the /skill-creator skill. Note however that here you're not actually creating a skill; you're performing a set of eval experiments. The methods of communicating graph structure are 1. verbally 2. via ASCII art 3. via JSON 4. via YAML 5. via mermaid I'd like you to investigate which of these result in faithfully communicating a graph to an LLM. The "graphs" will be "box-and-arrow" system architecture diagrams as commonly used in software engineering. You must investigate a range of complexities of these diagrams; we want to know where -- if anywhere -- along this complexity spectrum the ability of a particular method of representation breaks down in its ability to communicate the topology faithfully to the LLM. For ASCII art, JSON, and YAML you can invent the exact way in which the graph is represented; this choice should be something that a human author might realistically use while also aiming to be very effective at communicating the topology to the LLM. In other words, the choice should be aimed at giving those methods a good chance of showing that they can match or outperform purely verbal transmission, or transmission via a formal grammar intended for the purpose (mermaid).

Experiment Design

  • Subject model: Claude Sonnet 4.6 (temperature=1.0, 3 runs per condition)
  • Grader model: Claude Sonnet 4.6 (temperature=0, comparing against algorithmically-computed ground truth)
  • Formats: verbal prose, ASCII art, JSON, YAML, mermaid
  • Graphs: 5 software architecture diagrams at increasing complexity (3→22 nodes, 2→34 edges)
  • Questions per graph: 6 topology questions — forward neighbors, backward neighbors, reachability, sources, sinks, total degree
  • Total eval calls: 75 subject + 450 grading = 525 API calls

MEDIUM: E-commerce Backend

10 nodes, 14 edges — [Browser, CDN, API Gateway, Auth Service, Product Service, Order Service, Message Queue, Notification Service, Redis Cache, Postgres] 
This system is called E-commerce Backend. An e-commerce platform with caching, async processing, and a CDN.

The system consists of the following components: Browser, CDN, API Gateway, Auth Service, Product Service, Order Service, Message Queue, Notification Service, Redis Cache, Postgres.

Browser connects to: CDN (fetches static assets), API Gateway (sends API requests).
CDN forwards cache misses to API Gateway.
API Gateway connects to: Auth Service (authenticates requests), Product Service (routes product queries), Order Service (routes order mutations).
Auth Service reads user credentials to Postgres.
Product Service connects to: Postgres (reads product catalog), Redis Cache (caches product data).
Order Service connects to: Postgres (writes order records), Message Queue (publishes order events), Redis Cache (checks inventory cache).
Message Queue delivers order events to Notification Service.
Notification Service logs notification status to Postgres.

LARGE: Platform Architecture

16 nodes, 24 edges — [Mobile App, Web App, Load Balancer, API Gateway, Auth Service, User Service, Order Service, Payment Service, Search Service, Analytics Service, Event Bus, Notification Service, Stripe API, Elasticsearch, PostgreSQL, Redis] 
This system is called Platform Architecture. A full platform with event-driven processing, search, analytics, and external integrations.

The system consists of the following components: Mobile App, Web App, Load Balancer, API Gateway, Auth Service, User Service, Order Service, Payment Service, Search Service, Analytics Service, Event Bus, Notification Service, Stripe API, Elasticsearch, PostgreSQL, Redis.

Mobile App sends requests to Load Balancer.
Web App sends requests to Load Balancer.
Load Balancer distributes traffic to API Gateway.
API Gateway connects to: Auth Service (authenticates), User Service (routes user ops), Order Service (routes order ops), Search Service (routes search queries).
Auth Service connects to: Redis (checks session cache), PostgreSQL (reads credentials).
User Service connects to: PostgreSQL (reads/writes user profiles), Event Bus (publishes user events), Redis (caches profiles).
Order Service connects to: PostgreSQL (reads/writes orders), Payment Service (initiates payments), Event Bus (publishes order events), Redis (caches order status).
Payment Service connects to: Stripe API (processes charges), Event Bus (publishes payment events).
Search Service connects to: Elasticsearch (queries index), Redis (caches results).
Analytics Service writes analytics data to PostgreSQL.
Event Bus connects to: Notification Service (delivers events), Analytics Service (delivers events).
Notification Service logs notifications to PostgreSQL.

Findings

  1. Mermaid is the only format achieving 100% across all complexities. Zero failures in 90 graded answers. A formal grammar designed for graph description produces unambiguous syntax that LLMs parse reliably.
  2. JSON and YAML are excellent and statistically indistinguishable. 100% everywhere except Q5 (sinks) at medium complexity, where each has a single failure in 3 runs. The failure is a reasoning issue (negative universal quantification), not a format comprehension issue.
  3. Verbal prose is surprisingly robust. Competitive with structured formats, failing only on exhaustive enumeration questions (sinks, sources) at higher complexity. Natural language edge descriptions are well-understood by LLMs.
  4. ASCII art degrades significantly from ~10 nodes onward. Three failure modes:
    • Arrow direction confusion — the model misreads which way arrows point in dense layouts (Q2 backward neighbors: 0/3 at medium)
    • Global scan failures — cannot reliably identify sinks by scanning a 2D layout (Q5: 0/3 at medium and large)
    • Author-side scalability — the hand-crafted XL ASCII art had a missing edge in the first run, showing the format has a completeness problem for authors too
  5. The complexity threshold is ~10 nodes / 14 edges. Below this, all formats are perfect. At medium (10/14), ASCII drops to 61% while others remain at 94–100%.
  6. Q5 (sinks) is universally the hardest question. Requires verifying that a node never appears as a source in any edge — negative universal quantification. Only mermaid achieves 100% on this question across all complexities.

Practical Recommendations

  • For LLM consumption, prefer mermaid or JSON/YAML edge lists over ASCII diagrams
  • ASCII art may still be useful as a supplement alongside a structured format
  • For simple graphs (≤6 nodes), any format works — don’t over-engineer the representation
  • If using verbal descriptions for larger graphs, structure them carefully: one edge per sentence, explicit directionality

Methodology Notes

  • ASCII art was hand-crafted (not programmatically generated) to give the format its best chance. Real-world ASCII diagrams would likely perform worse.
  • Initial run revealed a missing edge in the XL ASCII diagram, requiring a second run with fixes. Both benchmark files are preserved in the results.
  • Grading was refined between runs to accept any valid path for reachability questions, not just the specific BFS-shortest path.
  • Temperature=1.0 for the subject model ensures variance between runs; temperature=0 for the grader ensures deterministic grading.
